# Swift BarnardCore Split

## Problem

The Swift package currently places deterministic protocol logic in the same
target as Apple-bound code. That prevents the reusable logic from compiling
for non-Apple Swift targets and makes it difficult to prove that protocol
outputs are independent of Foundation, Security, CoreBluetooth, and UIKit.

## Goals

- Publish a `BarnardCore` Swift product containing deterministic RPID, ENIN,
  signing, and policy logic.
- Keep `BarnardCore` limited to the Swift standard library.
- Express byte-oriented Core inputs and outputs as `[UInt8]` and time values as
  integer Unix seconds.
- Inject randomness, key storage, and time through Core protocols.
- Keep the existing Apple-facing `Barnard` public API and behavior unchanged.
- Keep the Swift package and Flutter plugin copies byte-identical where their
  source is shared.

## Non-goals

- Changing Scan, Advertise, Central, Peripheral, GATT, or Transport behavior.
- Changing an on-wire payload, key derivation, signature, or policy decision.
- Moving CoreBluetooth state machines into `BarnardCore`.
- Adding an Android platform driver.
- Replacing the existing Flutter/CocoaPods integration with Swift Package
  Manager.

## Glossary

- **Core**: deterministic, platform-independent logic in the `BarnardCore`
  product.
- **Platform layer**: the `Barnard` product and Flutter plugin code that adapt
  Foundation and Apple services to Core.
- **RPID**: rolling proximity identifier.
- **ENIN**: exposure notification interval number.
- **Scan / Advertise / Central / Peripheral / GATT / Transport**: retain their
  existing Barnard meanings; this refactor does not change them.

## Architecture

`BarnardCore` owns:

- SHA-256, HMAC-SHA256, HKDF-SHA256, and AES-128 operations needed by Barnard;
- RPID, RPIK, TEK, display ID, event hash, payload, and ENIN calculations;
- deterministic secp256k1 key derivation, signing, recovery, and proof messages;
- deterministic v2 policy decisions;
- the dependency protocols `BarnardCoreRandomSource`,
  `BarnardCoreKeyStorage`, and `BarnardCoreClock`.

The Apple-facing `Barnard` target owns:

- Foundation `Data` and `Date` conversion;
- Security-backed random byte generation;
- `UserDefaults` key storage;
- CoreBluetooth, UIKit, and application lifecycle behavior;
- the existing public `BarnardEngine` and `BarnardIdentity` APIs.

The dependency direction is one-way:

```text
Barnard / Flutter Apple layer
            |
            v
        BarnardCore
            |
            v
   Swift standard library
```

The Flutter plugin compiles the same Core source files in its CocoaPods target.
`scripts/check-swift-mirror.sh` compares every shared Core and platform-adapter
file byte-for-byte.

## Observable behavior

For the same bytes, event code, ENIN configuration, and Unix time:

1. TEK derivation returns the same 16 bytes as before the split.
2. RPIK and RPID derivation return the same 16-byte values as before the split.
3. Event hashes, display IDs, and advertised payloads are byte-identical.
4. Signing private keys, compressed public keys, recoverable signatures, and
   proof messages are byte-identical.
5. ENIN boundary and v2 policy decisions are unchanged.
6. `BarnardEngine` and `BarnardIdentity` callers continue to use their existing
   Foundation-based API.

The committed fixed-input behavior vector is both the pre-split record and the
post-split regression oracle.

## Dependency behavior

- A caller that needs a new secret passes a `BarnardCoreRandomSource` and a
  `BarnardCoreKeyStorage` to `BarnardCoreKeyManager.loadOrCreate`.
- Existing stored bytes are returned without invoking the random source.
- Newly generated bytes are persisted through the injected storage protocol.
- Payload calculation reads Unix seconds from an injected
  `BarnardCoreClock`.
- Apple implementations of those protocols remain outside `BarnardCore`.

## Error and boundary behavior

- Existing invalid-length fallback and precondition behavior remains unchanged.
- ENIN lengths remain clamped to the existing 12-to-3,600-second range.
- A stable read that crosses an ENIN boundary still returns no value.
- No retry, timeout, or backoff behavior changes in this refactor.

## Compatibility

- `Barnard` remains the existing Apple-facing library product.
- Its generated public API must have no removals or signature changes.
- Existing `BarnardTests` remain unchanged and must pass.
- `BarnardCore` is additive. It exposes only standard-library shapes.
- There are no JSON Schema or other public serialized-shape changes, so
  `schema/barnard/v1` does not change.

## Security and privacy

The refactor does not change key derivation, persistence keys, signature
construction, or on-wire bytes. It does not add device-unique persistent
identifiers to any on-wire payload. Device secrets remain in the Apple platform
layer's existing local storage and are passed into Core only for deterministic
calculation.

## Example

```swift
import BarnardCore

struct FixedClock: BarnardCoreClock {
  let seconds: Int64

  func currentUnixSeconds() -> Int64 {
    seconds
  }
}

let secret = [UInt8](repeating: 0x11, count: 32)
let tek = BarnardCoreCrypto.deriveTekForEvent(
  deviceSecret: secret,
  eventCode: "event-42"
)
let payload = BarnardCoreCrypto.currentPayload(
  tek: tek,
  clock: FixedClock(seconds: 1_700_000_100)
)
```

The example performs no Foundation conversion and no platform service access.

## Acceptance evidence

- Run the committed behavior vector before and after the split and compare the
  recorded RPID, ENIN, signing, and policy values byte-for-byte.
- Run all existing Swift package tests unchanged.
- Build the `BarnardCore` target on the host and with the Android Swift SDK.
- Grep `Sources/BarnardCore` for Foundation, CryptoKit, CoreBluetooth,
  Security, UIKit, CommonCrypto, `Data`, `Date`, `UserDefaults`, and
  `SecRandom`; the search must return no matches.
- Compare the generated `Barnard` public API before and after the split.
- Run `scripts/check-swift-mirror.sh`; every shared source pair must match
  byte-for-byte.
